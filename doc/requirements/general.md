# 二、项目需求分析
## 2.1 引言

### 2.1.1 项目背景
随着人们生活水平的提高和交通的便利化，旅游已成为现代人休闲娱乐、增长见识的重要方式。然而，在旅行的全生命周期中，用户面临着诸多痛点：
1.  **规划繁琐**：制定详细的旅行计划需要查阅大量分散的攻略（小红书、马蜂窝等），并在地图软件（高德、百度）之间反复切换以确认路线，缺乏一个集成的协作平台。
2.  **协同困难**：多人结伴出游时，行程信息的同步往往依赖微信群聊，信息容易被淹没，缺乏统一的行程管理工具来实时更新和通知。
3.  **信息碎片化**：旅行结束后的经验分享往往分散在各个社交平台，且难以直接转化为他人的可执行行程。
4.  **智能辅助缺失**：传统的旅行应用缺乏基于上下文的智能推荐，用户在面对突发情况或需要个性化建议时，难以获得即时的智能帮助。

基于上述背景，本项目旨在开发一款“集成前端、服务器端及网络数据库的旅行信息分享应用”。该系统将整合行程规划、多用户协同、社交分享以及 AI 智能助手功能，为用户提供从“行前规划”、“行中管理”到“行后分享”的一站式解决方案。

### 2.1.2 定义与缩写
为了确保需求理解的一致性，本文档使用以下术语和缩写：

*   **系统/应用**：指代待开发的“旅行信息分享应用”。
*   **用户**：注册并登录本系统的终端使用者，包括普通旅行者和内容创作者。
*   **行程**：用户创建的旅行计划集合，包含目的地、日期范围、每日日程、具体活动及交通路线等信息。
*   **活动**：行程中的最小单元，指具体的游玩景点、餐饮或住宿安排。
*   **RAG（检索增强生成）**：检索增强生成技术，指系统通过检索本地数据库中的行程和帖子数据，辅助大语言模型生成更准确回答的技术。
*   **DDD（领域驱动设计）**：领域驱动设计，一种软件开发方法论，本项目要求采用该思想进行架构设计。
*   **POI（兴趣点）**：兴趣点，指地图上的具体地点信息。

## 2.2 项目概述

### 2.2.1 产品描述
本项目产品定位为一款**基于 Web 的社交化旅行协同管理平台**。它不是单一的游记博客，也不是纯粹的地图导航工具，而是一个连接“人、行程、内容”的综合性系统。

从系统架构角度来看，产品将采用 B/S（浏览器/服务器）架构：
*   **前端**：提供响应式的用户界面，支持 PC 端和移动端浏览，强调交互流畅性和视觉美观性。
*   **后端**：提供 RESTful API 接口处理业务逻辑，并维持 WebSocket 连接以支持实时通讯。
*   **数据层**：利用关系型数据库存储结构化的用户、行程和社交数据。

产品核心理念是**“数据互通”**：用户的行程规划可以直接转化为社交平台的分享内容（游记），而社交平台上的优质内容（他人游记）也可以被引用或参考用于新的行程规划，AI 助手则贯穿其中，利用这些互通的数据提供智能服务。

### 2.2.2 产品功能
根据业务需求，系统需具备以下五大核心功能模块：

#### 1. 用户认证与管理模块
*   **注册与登录**：支持通过邮箱或用户名注册新账号，并提供安全的登录验证机制。
*   **个人资料管理**：允许用户设置头像、个人简介、常驻地等信息，展示用户的个人主页。
*   **安全机制**：必须对用户密码进行加密存储，并使用 Token 机制维持会话状态。

#### 2. 旅行核心模块
*   **行程创建与编辑**：用户应能创建多日行程，设置预算、起止日期和可见性（公开/私密）。
*   **日程管理**：支持按“天”维度的日程编排，用户可在每一天中添加、删除、排序具体的活动（Activity）。
*   **路线规划与集成**：系统需集成第三方地图服务（如高德地图），根据用户添加的活动地点，自动计算活动之间的交通方式、距离和预估耗时，并展示在行程视图中。
*   **多人协同**：支持行程创建者邀请其他用户加入行程，赋予成员编辑权限，实现多人共同规划同一份行程。

#### 3. 社交互动模块
*   **社区动态**：用户可以将结束的行程发布为图文并茂的游记（Post），支持上传多张图片和添加标签。
*   **互动体系**：支持用户对帖子进行点赞、评论，以及关注感兴趣的创作者。
*   **即时通讯**：系统应内置即时聊天功能，支持好友间的私聊以及行程小组内的群聊，用于行进过程中的实时沟通。

#### 4. AI 智能助手模块
*   **智能问答**：集成大语言模型（如 DeepSeek），为用户提供旅行建议、景点介绍等自然语言问答服务。
*   **知识增强（RAG）**：AI 应当能够“读取”系统内部的公开行程和热门帖子，当用户询问特定目的地时，能结合站内真实数据给出回答，而非仅依赖模型通用知识。

#### 5. 后台管理模块
*   **资源管理**：管理员应能查看系统内的用户列表、行程统计和帖子内容，具备对违规内容进行处理的权限。

### 2.2.3 用户特点
本系统的预期用户群体主要包括：

1.  **旅行爱好者 (核心用户)**：
    *   **特点**：乐于探索新目的地，习惯使用互联网工具规划行程，有强烈的分享欲和社交需求。
    *   **需求**：操作简便，移动端体验好，能快速从他人的游记中“抄作业”。
2.  **团队/家庭出游者**：
    *   **特点**：通常以 2-5 人的小组形式出现，需要协调多人的意见和时间。
    *   **需求**：强烈的协同编辑需求，以及实时的群组沟通工具。
3.  **内容创作者**：
    *   **特点**：擅长摄影和写作，希望通过分享获取关注。
    *   **需求**：美观的内容编辑器，高效的图片上传功能。
4.  **系统管理员**：
    *   **特点**：负责维护系统秩序。
    *   **需求**：清晰的数据监控面板和管理接口。

### 2.2.4 运行环境
为了保证系统的可访问性和性能，拟定的运行环境要求如下：

*   **客户端环境**：
    *   主流 Web 浏览器：Google Chrome, Microsoft Edge, Safari, Firefox（建议最新版本）。
    *   网络环境：需具备互联网连接，以访问地图 API 和 AI 服务。
*   **服务器端环境**：
    *   操作系统：跨平台支持（Windows/Linux/macOS），生产环境建议 Linux。
    *   运行环境：Python 3.8 及以上版本。
*   **数据库环境**：
    *   MySQL 8.0 及以上版本，支持 InnoDB 存储引擎以保障事务完整性。

### 2.2.5 实验目的
本项目作为“数据库系统实践”及相关课程的综合设计项目，其核心目的在于：

1.  **全栈开发实践**：验证学生整合前端（React）、后端（Flask）及数据库技术的综合开发能力。
2.  **数据库设计应用**：实践 E-R 建模、范式设计以及 SQL 查询优化，验证数据库理论在实际复杂业务场景中的应用。
3.  **架构设计落地**：探索并实践领域驱动设计（DDD）在 Web 应用中的落地，理解分层架构与代码解耦的重要性。
4.  **前沿技术融合**：尝试将生成式 AI（LLM）与传统 Web 业务系统进行有机结合（RAG），探索智能化应用的设计模式。
